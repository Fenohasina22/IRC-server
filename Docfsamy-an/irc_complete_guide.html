Foundations

1. Networks & IP Addresses

A network is simply a collection of computers that can send data to each other. The internet is the biggest network — billions of computers all connected together.

For computers to find each other, every device on a network gets an IP address — a unique numerical label, like a home address for your computer.

IPv4 address

Four numbers 0–255 separated by dots.
192.168.1.42
About 4 billion possible addresses. Running out.

IPv6 address

Eight groups of hex digits.
2001:db8::1
340 undecillion possible addresses. Won't run out.

🏠
Analogy

An IP address is like a street address. If you want to send a letter to someone, you need their address. If you want to send data to a computer, you need its IP address.

Special addresses

AddressMeaning
127.0.0.1Localhost — your own machine. Sending to this address talks to yourself. Used for testing.
0.0.0.0All interfaces — when binding a server, means "listen on every network interface of this machine".
255.255.255.255Broadcast — send to all devices on the local network at once.

2. Ports

An IP address identifies a machine. But a machine runs many programs at once — a web server, an IRC server, a game server. How does incoming data know which program it's for? That's what ports solve.

A port is a number from 0 to 65535. Think of it as the "apartment number" inside the building (the IP address). A connection is identified by the combination: IP:port.

🏢
Analogy

The IP address is the building. The port is the apartment number. 192.168.1.10:6667 means building 192.168.1.10, apartment 6667.

PortUsed for
80HTTP (web)
443HTTPS (secure web)
22SSH
6667IRC (the default IRC port)
6697IRC over TLS (secure)
Port ranges

Ports 0–1023 are "well-known" and require root to use. Ports 1024–49151 are "registered". Ports 49152–65535 are ephemeral — the OS assigns these randomly to clients when they connect out. Your ft_irc listens on a port you choose (usually above 1024 for testing).

3. What is a Socket?

A socket is the programming interface the OS gives you to use the network. It's an endpoint — one side of a two-way communication channel.

From your program's perspective, a socket is just an integer — a file descriptor (fd). Once a connection is open, you read from it and write to it exactly like a file. The OS handles all the networking underneath invisibly.

📞
Analogy

A socket is like a phone. Before you talk, both sides need a phone (socket). One side dials (connect), the other answers (accept). Once connected, you speak and listen freely. When done, both sides hang up (close). The phone company (OS kernel) handles everything in between.

What the OS handles for you

You never touch raw network packets. The OS kernel does all of this transparently:

  • Breaking your data into packets small enough to travel the network
  • Numbering packets so the receiver can reassemble them in order
  • Waiting for acknowledgements and retransmitting lost packets
  • Buffering incoming data until your program calls recv()

4. TCP vs UDP

There are two main transport protocols. TCP is what IRC uses — it's worth understanding why.

TCP — Transmission Control Protocol

Reliable, ordered, stream-based.

Every byte you send is guaranteed to arrive, in the same order you sent it. If a packet is lost, TCP retransmits it. There's a "connection" — both sides acknowledge the link is open.

Used for: web (HTTP), email, IRC, SSH — anything where data must arrive correctly.

UDP — User Datagram Protocol

Fast, unreliable, message-based.

Fire-and-forget. Packets may arrive out of order or not at all — UDP doesn't care. No connection setup, just send.

Used for: video streaming, games, DNS — anything where speed matters more than perfect delivery.

Why IRC uses TCP

IRC is a chat protocol. If a message is lost, that's unacceptable — the user needs to see every message. TCP's reliability guarantees are exactly what chat needs. The slight latency overhead doesn't matter at human conversation speeds.

The "stream" nature of TCP

This is the single most important thing to understand for your server. TCP sends a continuous stream of bytes — it has no concept of "messages". If you send "NICK alice\r\nUSER bob\r\n" in one call, the receiver might get all of it, or "NICK ali" then "ce\r\nUSER bob\r\n", or three separate chunks. The data always arrives complete and in order — but not necessarily in the same "chunks" you sent.

This is why you need a receive buffer

You cannot assume one recv() call gives you one complete IRC message. You must accumulate bytes in a per-client buffer and scan for \r\n yourself. More on this in section 8.

6. The Server Loop

A server needs to handle many clients simultaneously. You set up one listening socket that accepts new connections, and then monitor all connected clients for incoming data. This is the fundamental loop every server follows.

  1. socket() — Create a new socket. The OS returns a file descriptor (an integer like 3).

  2. setsockopt(SO_REUSEADDR) — Allow reusing the port immediately after server restart. Without this, you'd wait ~2 minutes before binding the same port again.

  3. bind() — Attach the socket to an IP address and port. This is how the OS knows which port this socket "owns". Example: bind to 0.0.0.0:6667.

  4. listen() — Tell the OS "start accepting connections on this socket". The OS will queue incoming connections (up to backlog count) even before you call accept().

  5. poll() loop — Wait for activity on any monitored fd. When poll() returns, check which fds are ready.

  6. accept() — When the listening fd is ready, accept a new connection. Returns a brand new fd for that specific client. Add it to your poll set.

  7. recv() / send() — When a client fd is ready, read from it or write to it.

  8. close() — When client disconnects, close their fd and clean up.

// Pseudocode — the core structure of your server
int listen_fd = socket(AF_INET, SOCK_STREAM, 0);
setsockopt(listen_fd, SOL_SOCKET, SO_REUSEADDR, ...);
bind(listen_fd, &addr, sizeof(addr));
listen(listen_fd, 128);

while (running) {
    int ready = poll(fds, nfds, -1);   // block until something happens

    if (fds[0].revents & POLLIN)         // listening fd got a connection
        accept_new_client();

    for (each client fd with POLLIN)
        read_and_handle(client);           // recv → parse → respond

    for (each client fd with POLLHUP)
        disconnect_client(client);         // cleanup
}

7. poll() Explained

poll() is the mechanism that lets one thread watch many file descriptors at once. Without it, you'd have to either spin-check every fd constantly (wasting CPU) or block on one fd and miss events on others.

You give poll() an array of struct pollfd, each containing a file descriptor and what events you're interested in. poll() blocks until at least one fd has an event, then fills in the revents field for each ready fd.

struct pollfd {
    int   fd;       // the file descriptor to watch
    short events;   // what you're asking for: POLLIN, POLLOUT
    short revents;  // what actually happened (filled by the OS)
};
FlagMeaning
POLLINData is available to read (recv() won't block)
POLLOUTSocket is ready to write (send() won't block)
POLLHUPThe other side closed the connection
POLLERRAn error occurred on this fd
poll() — watching multiple clients

Click a client to simulate it sending data or disconnecting.

poll() vs select() vs epoll()

select() does the same job but has a hard limit of 1024 fds and a clunky API — avoid it. epoll() (Linux) and kqueue() (macOS) are more scalable at 10,000+ fds but overkill for ft_irc. poll() is the right tool for your use case.

8. Stream Buffering

Because TCP is a byte stream with no message boundaries, every client needs its own receive buffer — a string where you accumulate incoming bytes until you see a complete \r\n terminated line.

recv() buffer — what you might actually receive
Client is sending: NICK alice\r\nUSER bob 0 * :Bob\r\n 
// Per client — run this every time poll() says POLLIN
char tmp[512];
int n = recv(client.fd, tmp, 512, 0);
if (n == 0) { disconnect(client); return; }  // connection closed
if (n  < 0) { handle_error(); return; }

client.recv_buf.append(tmp, n);   // accumulate bytes

size_t pos;
while ((pos = client.recv_buf.find("\r\n")) != string::npos) {
    string line = client.recv_buf.substr(0, pos);
    client.recv_buf.erase(0, pos + 2);  // consume including \r\n
    parse_and_dispatch(client, line);    // handle the complete command
}
IRC Protocol

9. What is IRC?

IRC — Internet Relay Chat — is a text-based chat protocol invented in 1988 by Jarkko Oikarinen. It's defined in RFC 1459 (1993) and refined in RFC 2812 (2000).

The concept is simple: users connect to a server, identify themselves with a nickname, join channels (chat rooms), and exchange text messages — publicly in channels or privately between users.

Key concepts

Nick — your username on the server (e.g. alice)
Channel — a chat room (e.g. #general)
Operator — a privileged user in a channel
Server — the program you're building

What makes it special

IRC is plain text — every message is human-readable. There's no binary encoding, no compression. You can literally use nc (netcat) as a client and type commands by hand.

Try it with netcat

Once your server is running: nc 127.0.0.1 6667 — then type NICK test and USER test 0 * :Test User and press Enter after each. You're now hand-typing IRC!

IRC Protocol

10. IRC Message Format

Every IRC message is one line, terminated with \r\n, maximum 512 bytes total. The format has up to four parts:

:nick!user@host PRIVMSG #channel :Hello world\r\n
Prefix (optional) — who sent it
Command — what to do
Parameters — space-separated
Trailing — starts with :, can contain spaces

Parsing rules

  1. If the line starts with :, everything up to the first space is the prefix. Strip it.

  2. The next word (uppercase) is the command — e.g. PRIVMSG, JOIN, NICK.

  3. Split the rest by spaces to get parameters. Stop splitting when you see a token starting with : — that's the trailing parameter, and it gets everything to the end of the line (including spaces).

The prefix in detail

The prefix :nick!user@host contains three things separated by ! and @:

PartExampleMeaning

When your server broadcasts messages to other clients (e.g. forwarding a PRIVMSG), it prepends the sender's prefix so recipients know who sent it.

11. Client Registration Flow

A client is not considered connected in the IRC sense until they complete registration. Until then, your server should reject most commands with 451 ERR_NOTREGISTERED.

Registration requires receiving three commands: PASS (optional), NICK, and USER. The order of NICK and USER is flexible — register the client once both have been received.

The welcome burst

Once registration is complete, the server must send a sequence of numeric replies called the welcome burst:

// All replies take the form: :server_name CODE client_nick :message
":irc.server 001 alice :Welcome to the IRC network alice!alice@host"
":irc.server 002 alice :Your host is irc.server, running version 1.0"
":irc.server 003 alice :This server was created ..."
":irc.server 004 alice irc.server 1.0 o o"
":irc.server 375 alice :- irc.server Message of the day -"
":irc.server 372 alice :- Welcome! -"
":irc.server 376 alice :End of /MOTD command."
State tracking per client

Each client struct needs boolean flags: has_nick, has_user, registered, and optionally authenticated (if your server has a password). Check these on every command — reject unregistered clients early.

12. Channels

Channels are the core of IRC. A channel is a named chat room, prefixed with #. They are created automatically when the first user joins and destroyed automatically when the last user leaves.

JOIN and PART

// Client joins a channel
JOIN #general
JOIN #secret secretpass    // join with key (+k mode)

// Server must respond to a successful JOIN:
":alice!alice@host JOIN #general"     // broadcast to all in channel
":server 332 alice #general :Topic here"  // topic (if set)
":server 353 alice = #general :@alice bob" // NAMES list
":server 366 alice #general :End of NAMES" // end of NAMES

// Client leaves
PART #general :Goodbye!

PRIVMSG and NOTICE

// Send a message to a channel
PRIVMSG #general :Hello everyone!

// Send a private message to a user
PRIVMSG alice :Hey, got a minute?

// Server must NOT echo the message back to the sender
// Forward to all OTHER members of the channel with sender's prefix:
":bob!bob@host PRIVMSG #general :Hello everyone!"  // sent to alice, carol, etc.
The first user to JOIN becomes operator

When a channel is created (first JOIN), that user automatically receives operator status (@). Operators can use MODE, KICK, INVITE, and TOPIC.

TOPIC

// Set the topic
TOPIC #general :Welcome to the main channel!

// Query the topic
TOPIC #general

// Response: 332 if topic exists, 331 if no topic
":server 332 alice #general :Welcome to the main channel!"

NAMES and WHO

// List users in a channel
NAMES #general

// Response: 353 lists users, 366 ends the list
// @ prefix = operator, no prefix = regular user
":server 353 alice = #general :@alice bob @carol"
":server 366 alice #general :End of /NAMES list"

PING / PONG — the keep-alive

IRC clients can go idle. The server periodically sends a PING and expects a PONG back within a timeout (typically 60–120s). If no reply, the server disconnects the client. Your server should also respond to PING from the client.

// Server → Client
":server PING :1234567890"

// Client → Server (must echo the exact token)
"PONG :1234567890"

14. Modes

Modes are flags that modify the behavior of a channel (or user). The MODE command sets them with + (add) or - (remove). Only channel operators can set channel modes.

ModeNameEffectParameter?
+iinvite-onlyOnly invited users can JOINNo
+ttopic lockOnly operators can change the TOPICNo
+kkey (password)JOIN requires the correct keyYes — the key itself
+ooperatorGrant/revoke operator status to a userYes — the nickname
+luser limitMaximum number of members in the channelYes — the number

MODE parsing — the tricky part

The MODE command can combine multiple flags in one call. You iterate the mode string character by character, track whether you're in + or - context, and consume parameters from the argument list as needed.

// Examples
MODE #chan +i           // set invite-only (no param)
MODE #chan +k secret    // set key to "secret" (one param)
MODE #chan +o alice     // make alice an operator (one param)
MODE #chan +ik secret   // invite-only + key in one shot
MODE #chan -k           // remove key
MODE #chan +l 50        // limit to 50 users
MODE #chan -l           // remove limit

// After successful MODE, broadcast to all channel members:
":alice!alice@host MODE #chan +i"

// Parsing loop pseudocode:
bool adding = true;  // + by default
int param_idx = 0;
for (char c : mode_string) {
    if (c == '+') { adding = true; }
    else if (c == '-') { adding = false; }
    else { apply_mode(c, adding, params[param_idx++]); }
}

16. Signals & Clean Shutdown

When the user presses Ctrl+C or the OS sends SIGTERM, your server should shut down cleanly: send an ERROR to all clients, close all sockets, free memory.

The key rule: never do complex work inside a signal handler. Signal handlers can interrupt your code at any point, including in the middle of a malloc(). Doing anything non-trivial there causes undefined behavior.

// The safe pattern:
volatile sig_atomic_t g_running = 1;

void signal_handler(int sig) {
    g_running = 0;  // ONLY this. Nothing else.
}

int main() {
    struct sigaction sa = {};
    sa.sa_handler = signal_handler;
    sigaction(SIGINT, &sa, NULL);
    sigaction(SIGTERM, &sa, NULL);

    while (g_running) {
        int n = poll(fds, nfds, 1000);  // 1s timeout to recheck g_running
        if (n < 0 && errno == EINTR) continue;  // poll was interrupted by signal
        // ... handle events ...
    }

    // Cleanup happens HERE, in normal code, not in the signal handler
    for (auto& client : clients)
        send(client.fd, "ERROR :Server shutting down\r\n", ...);
    // close all fds, free memory...
}
volatile sig_atomic_t

This type is guaranteed to be readable/writable atomically even when interrupted by a signal. The volatile keyword prevents the compiler from caching the value in a register — it forces a fresh memory read every iteration of the loop.

17. The Data Model

Your server needs two core data structures. They cross-reference each other because a Client knows which channels it's in, and a Channel knows which clients are members.

struct Client {
    int         fd;           // the socket file descriptor
    string      nick;         // current nickname
    string      username;     // from USER command
    string      realname;     // from USER command
    string      hostname;     // their IP address
    bool        registered;   // has_nick && has_user (and auth if required)
    string      recv_buf;     // partial incoming data
    string      send_buf;     // data waiting to be sent
    set<Channel*> channels;   // channels this client is in
};

struct Channel {
    string      name;         // e.g. "#general"
    string      topic;
    set<Client*> members;    // all users in channel
    set<Client*> operators;  // users with @ (op) status
    set<string>  invite_list; // nicks allowed in (+i mode)
    string      key;          // password (+k mode)
    int         limit;        // max members, 0 = none (+l mode)
    bool        invite_only;  // +i
    bool        topic_locked; // +t
};

struct Server {
    int                     listen_fd;
    map<int, Client*>       clients;   // fd → client
    map<string, Channel*>   channels;  // name → channel
    vector<pollfd>          poll_fds;
};

Build order recommendation

1. TCP server loop with poll() and accept() → 2. recv buffer + line extraction → 3. NICK+USER registration + welcome burst → 4. PING/PONG → 5. JOIN+PART+PRIVMSG → 6. NAMES+TOPIC → 7. MODE → 8. KICK+INVITE → 9. Signal handling. Test each step with nc before moving on.
